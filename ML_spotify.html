Spotify Million Playlist Dataset

The Spotify dataset contains one million user-generated playlists with over 2 million unique tracks by nearly 300,000 artists. These playlists were created by US Spotify users between January 2010 and November 2017. The dataset is stored in a ZIP file, in which data have been split into multiple json slices. Each json file contains exactly 1,000 playlists and are named with following convention: mpd.slice.{start_playlist_ID}-{end_playlist_ID}.json.

import pandas as pd
import json
import fnmatch
from tqdm import tqdm
from zipfile import ZipFile

def load_zip_data(zip_file: str, number_of_files: int) -> list[dict]:
    """Load a specified number of json files from a zip file and return a list of collected playlists."""
    with ZipFile(zip_file) as zipfiles: # open ZIP file
        file_list = zipfiles.namelist()

        json_files = fnmatch.filter(file_list, "*.json")
        json_files = [file for index,file in sorted([(int(filename.split('.')[2].split('-')[0]), filename) for filename in json_files])]
        playlists: list[dict] = []

        for filename in tqdm(json_files[:number_of_files]): # for each json file
            with zipfiles.open(filename) as json_file:
                current_slice = json.loads(json_file.read())
                playlists.extend(current_slice['playlists']) # add new playlists

        return playlists

Defining baseline methods

Before looking at the architecture of the neural network, we will first define two baseline methods to compare against. These methods should get k track URIs as input and return a possible next song that could work with the initial playlist.

We will rapidly test two baseline methods in this section on a random playlist and analyze the result. We will then evaluate these baseline methods with all other methods by using k-gramm and analyze the playlist generation problem as a binary problem. There will be several evaluation cases.

Get next song based on the previous k songs with tracks occurrences

This method will analyse every playlist and find a song that frequently occurs after the k previous songs. In this method the order matters and only songs placed after the input songs will be returned. Moreover, The order of input songs also matters, which is why it could be possible that this method does not return or find a next song. For unseen combination of songs, this method will be a lower bound tobeat.

This method is far slower than the first one, but we expect that it will only return relevant songs, when they exist. We tried to improve the readibility with the apply and map methods from Dataframe to filter the relevant playlists, but the optimized version was 2 times slower than this one. Excecution times are long for this method when evaluating 20% of the 20.000 loaded playlists.

# exec on every opening
import pandas as pd

def get_next_song(song_uris: list[str]) -> str | None:
    """select the song that appears in together in the largest number of playlists with the previous songs."""
    song_occurrences: dict[str, int] = {}
        
    for tracks in playlists['tracks']:
        track_uris = [track['track_uri'] for track in tracks]
        
        if not all(song in track_uris for song in song_uris): # check if every songs are contained
            continue
            
        songs_count = 0
        
        for track_uri in track_uris:
            if (songs_count < len(song_uris) and
                track_uri == song_uris[songs_count]):
                songs_count += 1
                continue
            if songs_count < len(song_uris) or track_uri in song_uris:
                continue
            
            # current track_uri is not in song_uris and occurs after the k previous songs
            if track_uri in song_occurrences.keys():
                song_occurrences[track_uri] += 1
            else:
                song_occurrences[track_uri] = 1
    next_songs = [ track_uri for (track_uri, _n) in sorted(song_occurrences.items(), key=lambda item:item[1], reverse=True)] # sort by playlist count desc
    
    return next_songs[0] if len(next_songs) > 0 else None # may not found a next song

# method to analyse the occurences of a particular song in the playlists
def song_occurrence(song_uri: str, playlists: pd.DataFrame) -> int:
    """Return occurrence of one song in the playlists"""
    song_occurrence = 0
        
    for tracks in playlists['tracks']:
        track_uris = [track['track_uri'] for track in tracks]
        
        if song_uri in track_uris : # check if the song is contained
            song_occurrence += 1
    
    return song_occurrence

Defining a track embedding

A one-hot-encoded vector for each tracks would lead to vectors with 263.469 dimensions, which is very high. This may pose challenges in terms of memory allocation and computational efficiency. That's why we decided to firstly compute a track embedding before designing the RNNs. We will train a track embedding with 300 dimensions to better store the differences between tracks. Additionnally, 300 is a common choice for the number of dimensions. We will at the end normalize the vectors to the unit-length, so that each dimension value will be contained in [-1, 1]. After normalization, some methods based on cosine simularity will be faster, but this operation is destructive and doesn't allow further training.

Process tracks context for track embeddings

The computed track embedding should acknowledge tracks as close to each other when:

  • tracks are from the same playlists. We suppose that tracks in the same playlists could have related genres or style.
  • tracks from the same album
  • tracks from the same artist

That is why we will generate a context list of tracks for each artists and for each album as additional information with the initial collected playlists. The tracks URIs of these new playlists will represent the words and the playlists represent the documents, so that we can use word2vec from gensim to find a vector representation of track URIs. Before that, we will convert each track_uri to an unique number, to reduce needed space for the model.

Train a Word2Vec model for our tracks

Word2Vec offers two methods: skip-gram and continuous bag of words(CBOW). We will use skip-gram, as it seems that skip-gram works well with infrequent word in the datasets (NLP 101: Word2Vec Skip-gram and CBOW, Google archive word2vec). The method will learn the word embedding from the words context, which is composed of the surrounding words. As we already saw with the baseline methods, some songs could only occur once in the initial first 20,000 playlists. Tracks could occur at least three times in the context list: at least once in the 20,000 playlists, in one album and in one artist playlist.

Based on the computed track context, a track embedding will be computed with 300 dimensions and a window size of 10 for the skip-gram method. Each element of the vector will be in [-1, 1] after the use of unit_normalize. We choose 300 for the vector dimension as it is a common vector size for word embeddings next to 100 and 200. With 300 dimensions, the model could store more details on the 263,469 collected unique tracks than with only 100 or 200 dimensions.

Helper methods for switching between tracks uris and vectors

Based on this track embedding, two helper methods will be defined which will allow to write clearer code when dealing with the RNNs.

# exec on every opening
import numpy as np

def convert_track_to_vec(track_uri: str) -> np.ndarray:
    """Convert the track uri to a dense vector"""
    return track2vec.wv[track2number[track_uri]]

def convert_vec_to_track(vector: np.ndarray) -> str:
    """Convert a dense vector to a track uri"""
    return number2track[track2vec.wv.similar_by_vector(vector, topn=1)[0][0]]

With the computed track embedding and the cosine similarity from this model, a new method can be defined for a playlist generation. As said previously, with skip-gram, we could get similar tracks wich do not necessarily occur after the k initial songs, but mostly with similar contexts (same playlists, same artirst and/or same album). It's probably possible that two or more similiar tracks don't occur together in a playlist, but have similar surrounding tracks. That's why this method should always return a track and be more flexible than get_next_song.

As specified in the official documentation the method most_similar from gensim Word2Vec will compute the mean of the projection weight vectors of the input track URIs and the vectors for each word (here numbers representing track URIs) in the model.

In[]:
# exec on every opening

def get_next_similar_song(song_uris: list[str]) -> str:
    """select the next similar song based on a computed embedding: track2vec."""
    encoded_playlist = [ track2number[song_uri] for song_uri in song_uris ] # track2vec uses track_numbers as vocabulary
    similar_songs = track2vec.wv.most_similar(encoded_playlist, topn = 1) # [(track_number, sim(track_number, vocabulary))]
    
    return number2track[similar_songs[0][0]]

Defining a neural network

We will now train 2 recurrent neural networks (RNNs) based on this track embedding. Each RNN will accept a matrix of $N * 10$ with N a natural number and will return a vector representing a track URI. All other defined methods do not have such a limitation for the size of the base playlist. Furthermore, we will use k grams with k = 10 as a window size for generating X and y lists for training, validation and test data. It is the same windows size as for our trained track embedding.

As evaluation metric, we attempted the precision from keras, but it has always returned 1 for each epoch, so that we finally decided to test the RNNs with the other methods in the evaluation section at the end of this notebook. Additionally, the accuracy wasn't an easy metric to analyse while training the models and was always over 55%. We will mostly analyse the loss values.

import pandas as pd

def playlist_k_gram(tracks: list, k: int, step: int = 1) -> tuple[list[list], list]:
    """Split a list of tracks in k grams: X[i] contains a sequence of k tracks, y[i] the k+1 track."""
    X: list[list] = []
    y: list = []
        
    for i in range(0, len(tracks) - k, step):
        X.append(tracks[i:i+k])
        y.append(tracks[i+k])
    
    return X, y

def playlists_k_gram(playlists: pd.Series, k: int, step: int = 1) -> tuple[list[list], list]:
    """Split the list of playlists in k grams: X[i] contains a sequence of k tracks, y[i] the k+1 track."""
    X: list[list] = []
    y: list = []
        
    for playlist in playlists:
        playlist_x, playlist_y = playlist_k_gram(playlist, k, step)
        X.extend(playlist_x)
        y.extend(playlist_y)
    
    return X, y

Designing two recurent neural networks (RNNs)

The two trained recurent neural networks will expect 10 embedded tracks as input and return a vector representing a track URI. They will contain a LSTM unit and use the tanh activation fonction in the last layer, as vector values are in [-1, 1]. This is exacly the output range of tanh. The only difference will be the used loss function: mean squared error (MSE) or cosine similarity. The first will focus on vectors close to each other in term of distance and the second on vectors pointing in the same direction in the vector space. The two models should at least return vectors which are possibly similar in regard of the computed track embeddings.

Computing a vector based on a list of other vectors is more like a regression problem. That's the reason why many loss functions for classification problems like categorical crossentropy, Hinge or KL Divergence were not adapted for our task. MSE is the most commonly used regression loss, so that we preferred it in comparison to mean absolute error (MAE). Additionnaly MAE is more redust to outlier, because outliers will be treated as corrupted data and have a small impact on the loss value. However, in our context, vector values are in [-1, 1] and outliers aren't anomalies, but tracks with higher values. As a result, MSE is a better choice than MAE or Huge loss as outliers must be considered in our playlist generation problem (source).

Finally, we will use RMSprop as our optimizer, since it is one of the most popular optimization algorithms used in deep learning. This optimizer uses an adaptive learning rate.

RNN 1 with MSE

The chosen loss function will be here MSE, to prediction a vector which will be close to the expected vector in terms of distances or maybe equal to it. So we are here more focused on exact values. MSE will return a value in [0, 1] and a good loss value should be close to 0.

from keras.models import Sequential

embedding_dim = 300 

def __predict_next_track(song_uris: list[str], model: Sequential) -> str:
    """Predict the next possible song with help of the selected RNN."""
    x = np.zeros((1, playlist_max_len, embedding_dim))
    embedded_songs = [ convert_track_to_vec(song_uri) for song_uri in song_uris]
    x[0] = np.array([embedded_songs])

    next_song = model.predict(x, verbose=0)[0]

    return convert_vec_to_track(np.asarray(next_song).astype('float64'))

def predict_next_song_v1(song_uris: list[str]) -> str:
    """Predict the next possible song with help of a RNN. Needs one or more windows of 10 song_uris as input. This method uses RNN 1."""
    if not model:

Evaluation of each method

We have already split our test data with track uris into lists of K-gramms with 20% for testing purpose. As short reminder, the methods to be tested are:

  • 2 baseline methods: These methods will be helpful to evaluate our trained models and to define possible lower and upper bounds.
    • get_random_song: this method should have a bad score, as their is a chance of $\frac{1}{|tracks|}$ to pick a valid song according to our first evaluation case. For the next two cases, the chance could be greater, but it should remain very low. As a consequence, our three advanced methods have to be better than this first baseline method.
    • get_next_song: this method needs to go through each playlist, but should have a better score than the random method. As our tests have already shown, the next track is not always the real next one in the playlist, but it's most often the case. Additionally, with an unseen combination of tracks the returned value could be None.
  • 3 advanced methods: These methods should always return a track, even for an unseen combination of input tracks.
    • get_next_similar_song: this method uses the trained Word2Vec embedding model and the cosinus similarity to pick the next song.
    • predict_next_song: this method uses the first RNN, which was trained with the track embeddings and MSE as loss function.
    • predict_next_song_v2: this method uses the second RNN, which was trained with the track embeddings and cosine similarity as loss function.

We will use the precision as metric for our evaluations. As reminder: $Precision = \frac{\#TP}{\#TP + \#FP}$. We will only have true positive (TP), when the predicted track is the expected track or when the track is valid in the point of view of the test, else it will be a false positive (FP). Finally, the possible result None from get_next_song will be threated as a false positive in our evaluation in case if it occurs.

Evaluations to perform:

  • Evaluation 1: Given the k previous songs, the predicted song has to be the direct next one.
  • Evaluation 2: given k songs, the predicted song has to occur in the same playlist after these k songs.
  • Evaluation 3: given k songs, the predicted song should have a common album or artist with the next possible songs, or to occur in the same playlist after these k songs.
  • Finally, we will test unseen combinations with three self-made playlists and make a small user study for evaluating our three advanced methods: track embeddings and the two RNNs. This last evaluation will take the form of a google form.

As the evaluation of each method could take a lot of time, we added the progression bar from tqdm to have a visual estimation of the progression. Last but not least, evaluation 2 is an extension of evaluation 1 and evaluation 3 is an extension of evaluation 2, which leads to more flexible evaluations.

def extended_playlist_k_gram(tracks: list[str], k: int, step: int = 1) -> tuple[list[list[str]], list[list[str]]]:
    """Split a list of tracks in k grams: X[i] contains a sequence of k tracks, y[i] the tracks after these k tracks."""
    X: list[list[str]] = []
    y: list[list[str]] = []
        
    for i in range(0, len(tracks) - k, step):
        X.append(tracks[i:i+k])
        y.append(tracks[i+k:]) # all tracks from i+k to the end of the list
    
    return X, y

def extended_playlists_k_gram(playlists: pd.Series, k: int, step: int = 1) -> tuple[list[list[str]], list[list[str]]]:
    """Split the list of playlists in k grams: X[i] contains a sequence of k tracks, y[i] the tracks after these k tracks."""
    X: list[list[str]] = []
    y: list[list[str]] = []
        
    for playlist in playlists:
        playlist_x, playlist_y = extended_playlist_k_gram(playlist, k, step)
        X.extend(playlist_x)
        y.extend(playlist_y)
    
    return X, y

def similar_values(pred_track_uri: str | None, possible_next_tracks: pd.DataFrame) -> bool:
    """Check if predicted track is in possible_next_songs or has a common artist or album as possible_next_songs."""
    if not pred_track_uri: # pred_track_uri could be none
        return False
    if possible_next_tracks['track_uri'].isin([pred_track_uri]).any(): # pred track in the playlist after the k tracks
        return True
    
    pred_track = tracks.loc[tracks.index == track2number[pred_track_uri]]

    # common artist or common album
    return (pred_track['artist_uri'].isin(possible_next_tracks['artist_uri']).any() or
            pred_track['album_uri'].isin(possible_next_tracks['album_uri']).any())


def similar_values_eval_method(X: list[list[str]],
                               y: list[pd.DataFrame],
                               method: Callable[[list[str]], str|None]) -> dict[str, int]:
    """Evalution method for playlist generation: check if predicted track is in possible_next_songs or has a common artist or album as possible_next_songs."""
    score = {

    for index in tqdm(range(len(X))):
        next_song = method(X[index])
        
        if similar_values(next_song, y[index]):
            score['true'] += 1
            
    return score

def get_track_uri(song_id: str) -> str:
    """Convert the song id to a valid track uri."""
    return f'spotify:track:{song_id}'

def generate_track_suggestion(song_uris: list[str], methods: list[Callable[[list[str]], str|None]]) -> list[str | None]:
    """Execute defined methods to generate suggestions for the next track of the playlist."""
    result: list[str | None] = []

    for method in methods:
        result.append(method(song_uris))
    
    return result

Here, our models have used the 10 tracks with the same weight by mostly computing a mean as in the Word2Vec model from Gensim. This last feedback / suggestion could be an important view to keep in mind when evaluating the grades, as it is indeed an important factor.

Conclusion
In this Jupyter notebook several methods of playlists generation were designed and evaluated: two baseline methods and three methods using neural networks. Because of local hardware limitations or evaluation time, we could only use 20,000 of the one millions playlists, so that not every tracks from the original database are covered by our models.


We firstly defined two baseline methods, which could be seen as the under and upper bounds. After that we designed a track embedding thanks to Gensim's Word2Vec model. Based on it, we then trained two recurrent neural networks with different loss functions. At the end, every method was automatically tested with three different evaluation cases. Finally, we made a small user study with a google form for evaluating a small amount of self-generated playlists. We used the embedding model and the two RNN models on unseen combination of tracks for generation. This study shows that at least for the three designed playlists, the suggestions were generally correct with a mean score above 4 or 5 over 10.


Finally, our computed models don't achieve a better precision than the second baseline method, but these models are more flexible on unseen combinations of tracks and are way faster when used on a large amount of data. The precision of our neural networks were really low. However, these methods are at least far better than randomness. Possible improvements of our trained methods could be to load more playlists from the original dataset, which will improve the context of each tracks, or use data augmentation (e.g. shuffle the k-grams) or overparameters turning (e.g. changing the value of embedding dimensions or k).
